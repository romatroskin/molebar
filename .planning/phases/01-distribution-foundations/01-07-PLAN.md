<objective>
Execute the canonical Phase 1 deliverable — the 0.0.1 → 0.0.2 round-trip smoke test on a clean Mac. Phase 1 cannot be marked complete until this passes. The procedure is documented step-by-step in `01-RESEARCH.md` §Smoke Test Plan (15 steps, 5 acceptance criteria) and the per-requirement test map in `01-VALIDATION.md`. This plan carries the tasks of *running* that procedure and *recording* the results.
</objective>

<context>
This plan is mostly non-autonomous — the executor (or human operator) must:
1. Tag and push `v0.0.1`, observe CI, confirm the release artifacts are live.
2. Install the resulting `.dmg` on a clean Mac (or a fresh macOS user account on the dev machine — that's sufficient for "clean").
3. Make a tiny popover-text change, tag `v0.0.2`, push, observe CI.
4. Trigger the in-app update from the running 0.0.1, confirm Sparkle's flow.
5. Run the negative test (corrupt appcast.xml).

The work product is **evidence**: the GitHub Release URLs, screenshots, Console exports, and a deviation log. Phase 1's verification step (`/gsd-verify-work 1`) reads that evidence to certify the phase complete.

Because this plan executes against the live `main` branch / public repo / GitHub Pages, every step is *real* (not a dry run). If a step fails, the executor stops, captures diagnostics, and either fixes forward (cheap fixes — typo, missing secret) or escalates back to Plans 01-06 if the failure indicates an upstream defect.
</context>

  <name>Task 5: Negative test — corrupt appcast.xml; confirm Sparkle refuses the update (DIST-06)</name>

    - .planning/phases/01-distribution-foundations/01-RESEARCH.md §Smoke Test Plan (step 15); §Failure Modes
  </read_first>
  <what-built>
    Task 4 verified the positive path. The negative path — Sparkle MUST refuse a tampered appcast — is what makes the EdDSA verification meaningful. Without this test, we don't know whether Sparkle is actually verifying or silently accepting anything.
  </what-built>
  <how-to-verify>
    On the dev machine, against the gh-pages branch:

    ```bash
    git fetch origin gh-pages
    git checkout gh-pages

    # Note the start timestamp for the deviation log:
    date -u +"%Y-%m-%dT%H:%M:%SZ"
    # Capture this; we'll need an end timestamp too.

    # Edit appcast.xml — pick the sparkle:edSignature value, change exactly ONE base64 character
    # (e.g., flip the first 'A' to 'B'). Save.

    git add appcast.xml
    git commit -m "test: deliberate appcast corruption for Sparkle negative test"
    git push origin gh-pages

    # Wait ~30s for GH Pages redeploy. Verify the corruption is live:
    sleep 30
    curl -s https://puffpuff.dev/molebar/appcast.xml | grep sparkle:edSignature
    # Should show the corrupted base64 (different from Task 4's snapshot).
    ```

    On a fresh 0.0.1 install (reinstall MoleBar-0.0.1.dmg on the clean Mac, OR use a different fresh user account, OR delete /Applications/MoleBar.app and reinstall):

    1. Click the menu bar icon → `Check for Updates…`.
    2. Sparkle MUST NOT prompt to install. Expected outcomes (either is acceptable):
       - "Update could not be loaded" / "An update is available but the signature is invalid" UI message, OR
       - Silent failure with a Console log entry: `EdDSA signature verification failed`

    3. Open Console.app, filter subsystem `Sparkle`. Confirm at least one `EdDSA signature verification failed` entry exists in the test window.

    Then restore the appcast (this window is destructive against the public appcast — keep it short, ~5 minutes):

    ```bash
    # Revert the corruption commit:
    git revert HEAD --no-edit
    git push origin gh-pages

    # Note the end timestamp:
    date -u +"%Y-%m-%dT%H:%M:%SZ"

    # Wait ~30s for GH Pages redeploy. Verify the appcast is back to valid state:
    sleep 30
    curl -s https://puffpuff.dev/molebar/appcast.xml | grep sparkle:edSignature
    # Should match Task 4's pre-corruption snapshot.
    ```

    Final positive smoke (verifying restoration): on yet another fresh 0.0.1 install, `Check for Updates…` should once again succeed. (If Task 4 already established the positive path, this re-verification can be skipped — note the skip in the deviation log.)

    Document the corruption window (start + end timestamps) in 01-07-SUMMARY.md so any third party who checked the appcast during the window understands what they observed.
  </how-to-verify>
  <resume-signal>
    During the corruption window: `curl -s | grep edSignature` returns the tampered string AND Sparkle on a fresh 0.0.1 refuses the update AND Console shows the EdDSA failure log entry. Post-restoration: appcast back to valid state. Both timestamps captured into 01-07-SUMMARY.md §Task 5 evidence.

    ```bash
    # 1. Repo history scan — no private key material ever committed
    git log --all -p | grep -E "BEGIN.*PRIVATE KEY|^edpr|p8\b" || echo "CLEAN: no key material in repo history"
    # Expect: "CLEAN: no key material in repo history"

    # 2. CI logs scan — no key material printed in any release.yml run logs (last 5 runs)
    for id in $(gh run list --workflow=release.yml --limit 5 --json databaseId -q '.[].databaseId'); do
      echo "=== Run $id ==="
      gh run view "$id" --log 2>/dev/null | grep -iE 'edpr|BEGIN PRIVATE|\bp8\b' && echo "WARNING run $id may contain key material" || echo "Run $id clean"
    done
    # Expect: "Run <id> clean" for each id (not "WARNING")

    # 3. GitHub secret scanning alerts — Push Protection should have caught any leak attempt
    gh api repos/romatroskin/molebar/secret-scanning/alerts | jq 'length'
    # Expect: 0
    ```

    If any of the three scans returns a non-empty / non-zero result, this is a P0 incident:
    - Rotate the affected secret IMMEDIATELY (regenerate Sparkle key, update Info.plist, ship a v0.0.3 to refresh users).
    - If a private key is in repo history, force-push to remove it (`git filter-repo`); coordinate with the user before any force-push.
    - Document the incident in `.planning/incidents/0001-secret-leak.md` with timeline and remediation.
