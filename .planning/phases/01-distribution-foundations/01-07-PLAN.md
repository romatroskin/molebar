<objective>
Execute the canonical Phase 1 deliverable — the 0.0.1 → 0.0.2 round-trip smoke test on a clean Mac. Phase 1 cannot be marked complete until this passes. The procedure is documented step-by-step in `01-RESEARCH.md` §Smoke Test Plan (15 steps, 5 acceptance criteria) and the per-requirement test map in `01-VALIDATION.md`. This plan carries the tasks of *running* that procedure and *recording* the results.
</objective>

<context>
This plan is mostly non-autonomous — the executor (or human operator) must:
1. Tag and push `v0.0.1`, observe CI, confirm the release artifacts are live.
2. Install the resulting `.dmg` on a clean Mac (or a fresh macOS user account on the dev machine — that's sufficient for "clean").
3. Make a tiny popover-text change, tag `v0.0.2`, push, observe CI.
4. Trigger the in-app update from the running 0.0.1, confirm Sparkle's flow.
5. Run the negative test (corrupt appcast.xml).

The work product is **evidence**: the GitHub Release URLs, screenshots, Console exports, and a deviation log. Phase 1's verification step (`/gsd-verify-work 1`) reads that evidence to certify the phase complete.

Because this plan executes against the live `main` branch / public repo / GitHub Pages, every step is *real* (not a dry run). If a step fails, the executor stops, captures diagnostics, and either fixes forward (cheap fixes — typo, missing secret) or escalates back to Plans 01-06 if the failure indicates an upstream defect.
</context>

<task id="01-07-01" type="manual">
<title>Tag and push v0.0.1; observe release.yml succeed</title>
<read_first>
- .planning/phases/01-distribution-foundations/01-RESEARCH.md (§Smoke Test Plan, steps 1-2)
- .planning/phases/01-distribution-foundations/01-VALIDATION.md (Per-Task Verification Map: DIST-04, DIST-05)
- .github/workflows/release.yml (the workflow that will fire on tag push)
</read_first>
<action>
Run on the dev machine, with the local repo already pushed to `origin/main`:

```bash
git tag v0.0.1
git push origin v0.0.1
```

Open `https://github.com/romatroskin/molebar/actions` in a browser. Wait for the most recent `release.yml` run triggered by `v0.0.1` to reach status `completed success` (typical wall-clock: 6–12 minutes on `macos-15`).

If the run fails:
- Capture the failing step name and error log excerpt.
- If failure is a known stub-not-yet-set guard (e.g., `MACOS_CERT == ''`) — that's expected and the workflow should `continue-on-error: false` ONLY for the unsigned-path steps; the guarded steps must `if`-skip cleanly without failing the job. Verify that's the case; if a guarded step is failing instead of skipping, escalate back to Plan 06 to fix the workflow.
- If failure is in an unguarded step (build, DMG, sign_update, gh-pages publish, GH Release upload), capture full step logs and stop here — do not delete the tag yet. Document, then escalate.

Then verify the release artifacts:

```bash
gh release view v0.0.1 --json assets --jq '.assets[].name'
# Expect output to include: MoleBar-0.0.1.dmg

curl -fsSI https://romatroskin.github.io/molebar/appcast.xml
# Expect: HTTP/2 200

curl -s https://romatroskin.github.io/molebar/appcast.xml | grep -c 'sparkle:version="0.0.1"'
# Expect: 1
```
</action>
<acceptance_criteria>
- `gh run list --workflow=release.yml --limit 1` shows the v0.0.1 run with status `completed success`
- `gh release view v0.0.1 --json assets --jq '.assets[].name'` output contains `MoleBar-0.0.1.dmg`
- `curl -fsSI https://romatroskin.github.io/molebar/appcast.xml` returns `HTTP/2 200`
- `curl -s https://romatroskin.github.io/molebar/appcast.xml | grep -c 'sparkle:version="0.0.1"'` returns `1`
- `curl -s https://romatroskin.github.io/molebar/appcast.xml | grep -c 'sparkle:edSignature'` returns at least `1`
</acceptance_criteria>
</task>

<task id="01-07-02" type="manual">
<title>Install MoleBar-0.0.1.dmg on a clean Mac (or fresh user account); confirm popover</title>
<read_first>
- .planning/phases/01-distribution-foundations/01-RESEARCH.md (§Smoke Test Plan, steps 2-6)
</read_first>
<action>
On a clean Mac (or a freshly-created macOS user account on the dev machine — that's sufficient):

1. Download the DMG: `curl -L -o MoleBar-0.0.1.dmg https://github.com/romatroskin/molebar/releases/download/v0.0.1/MoleBar-0.0.1.dmg`
2. Open the DMG: `open MoleBar-0.0.1.dmg`
3. Drag `MoleBar.app` from the mounted DMG into `/Applications/` (in Finder).
4. Eject the DMG: `hdiutil detach /Volumes/MoleBar`
5. Launch the app: `open /Applications/MoleBar.app`
6. Expected first-launch friction: macOS Gatekeeper shows "MoleBar.app is from an unidentified developer" because Phase 1 is unsigned. Resolve: System Settings → Privacy & Security → "Open Anyway" for MoleBar. **This is expected and documented in README; Phase 1.5 retrofits signing to eliminate it.**
7. Confirm the menu bar icon appears (no Dock icon — `LSUIElement = YES`).
8. Click the menu bar icon. Confirm popover shows: `MoleBar 0.0.1 — nothing here yet`, `Check for Updates…`, `Quit`.

Capture: a screenshot or visual description of the popover with the version string visible.
</action>
<acceptance_criteria>
- `defaults read /Applications/MoleBar.app/Contents/Info CFBundleShortVersionString` outputs `0.0.1`
- `defaults read /Applications/MoleBar.app/Contents/Info CFBundleIdentifier` outputs `app.molebar.MoleBar`
- `ls /Applications/MoleBar.app/Contents/Helpers/mole/mole` exists (the bundled Mole Shell wrapper)
- `pgrep -x MoleBar` returns a PID (app is running)
- Manual confirmation: popover renders with the three documented items; menu bar icon visible; no Dock icon
</acceptance_criteria>
</task>

<task id="01-07-05" type="manual">
<title>Negative test: corrupt appcast.xml; confirm Sparkle refuses the update</title>
<read_first>
- .planning/phases/01-distribution-foundations/01-RESEARCH.md (§Smoke Test Plan, step 15; §Failure Modes)
</read_first>
<action>
On the dev machine, against the gh-pages branch:

1. `git fetch origin gh-pages`
2. `git checkout gh-pages`
3. Open `appcast.xml`. Pick the `sparkle:edSignature` attribute value. Change exactly ONE base64 character (e.g., the first `A` becomes `B`). Save.
4. Commit: `git add appcast.xml && git commit -m "test: deliberate appcast corruption for Sparkle negative test"`
5. `git push origin gh-pages`
6. Wait ~30 seconds for GitHub Pages to redeploy the corrupted file. Verify with: `curl -s https://romatroskin.github.io/molebar/appcast.xml | grep sparkle:edSignature` (should now show the corrupted value).

On a fresh 0.0.1 install (re-install MoleBar-0.0.1.dmg on the clean Mac, or use a different fresh user account):

7. Click the menu bar icon → `Check for Updates…`.
8. Acceptance criterion 5: Sparkle does NOT prompt to install; instead either (a) shows "Update could not be loaded" / "An update is available but the signature is invalid" or (b) silently fails with a Console log entry `EdDSA signature verification failed`.

Open Console.app, filter `Sparkle` subsystem:

9. Confirm the EdDSA failure entry is present.

Restore the appcast:

10. `git checkout gh-pages -- appcast.xml` (or revert the typo commit and force-push gh-pages — coordinate with Plan 06's rules on gh-pages mutability).
11. `git revert <typo-commit-sha>` then `git push origin gh-pages`.
12. Verify the appcast is back to a valid state: `curl -s https://romatroskin.github.io/molebar/appcast.xml | sparkle-validation-script-from-Plan-06` (or manually inspect that the edSignature matches what `sign_update` would produce for that DMG).

This is a destructive test on the public appcast for ~5 minutes. Document the start and end timestamps so any users who happen to check during the window understand what they observed.
</action>
<acceptance_criteria>
- During the corrupted window, `curl -s https://romatroskin.github.io/molebar/appcast.xml | grep -c 'sparkle:edSignature='` shows the corrupted base64 string
- Sparkle on the fresh 0.0.1 install does NOT install the update (manual confirmation: no prompt, no version change)
- Console.app shows at least one `EdDSA signature verification failed` log entry from the Sparkle subsystem during the test window
- Post-restoration, `curl -s https://romatroskin.github.io/molebar/appcast.xml | grep sparkle:edSignature` returns a valid (non-corrupted) signature; a fresh `Check for Updates…` on a 0.0.1 install once again succeeds (verifying restoration)
</acceptance_criteria>
</task>

<task id="01-07-06" type="manual">
<title>Final OSS-04 secret-leak audit across the round-trip CI runs</title>
<read_first>
- .planning/phases/01-distribution-foundations/01-VALIDATION.md (Per-Task Verification Map: OSS-04 row)
- .planning/phases/01-distribution-foundations/01-RESEARCH.md (§Security Domain — Known Threat Patterns)
</read_first>
<action>
After all preceding tasks pass:

```bash
# 1. Repo history scan
git log --all -p | grep -E "BEGIN.*PRIVATE KEY|^edpr|p8\b" || echo "CLEAN: no key material in repo history"

# 2. CI logs scan for the v0.0.1 and v0.0.2 release runs
for id in $(gh run list --workflow=release.yml --limit 5 --json databaseId -q '.[].databaseId'); do
  echo "=== Run $id ==="
  gh run view "$id" --log 2>/dev/null | grep -iE 'edpr|BEGIN PRIVATE|\bp8\b' && echo "WARNING run $id may contain key material" || echo "Run $id clean"
done

# 3. Secret scanning alerts
gh api repos/romatroskin/molebar/secret-scanning/alerts | jq 'length' || echo "0 (or scanning not enabled — that's a Plan 01 regression)"
```

If any of the three scans return a non-empty / non-zero result, treat as a P0 incident — rotate the affected secret, force-push history if the repo is still small enough to safely rewrite (`git filter-repo` is the recommended tool), document the incident in `.planning/incidents/0001-secret-leak.md`, and escalate.

If all three scans are clean, capture the output as evidence in the deviation log under §OSS-04 audit.
</action>
<acceptance_criteria>
- `git log --all -p | grep -E "BEGIN.*PRIVATE KEY|^edpr|p8\b"` produces no matches
- For each of the last 5 release.yml runs, `gh run view <id> --log | grep -iE 'edpr|BEGIN PRIVATE|\bp8\b'` produces no matches
- `gh api repos/romatroskin/molebar/secret-scanning/alerts | jq 'length'` returns `0`
- A clean audit summary is written to the plan summary
</acceptance_criteria>
</task>
