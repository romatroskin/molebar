<objective>
Generate, secure, and bootstrap the FROZEN-after-first-release Sparkle EdDSA keypair (per D-05 / D-06 / D-07 / D-08 / Pitfall A2). This is the single most-critical cryptographic decision in the entire project: every prior MoleBar install will only accept updates signed by the matching private key, and there is no rotation path on this code path. The plan is split into three checkpoint tasks because each step is a non-automatable user-machine + GitHub-UI operation: (1) generate the keypair on the dev Mac into the login Keychain (per D-05), (2) replace the SUPublicEDKey placeholder in Info.plist with the real public key (per D-08), (3) export the private key once and paste it into GitHub Actions repo secrets (per D-07).

Purpose: unblocks Plan 06's release workflow (which signs DMGs via `sign_update --ed-key-file <secret>`) and Plan 07's smoke test (where a 0.0.1 install must EdDSA-verify a 0.0.2 update against the public key embedded in v0.0.1). Locks D-08 — once a v0.0.1 release ships from Plan 06, the public key in v0.0.1's Info.plist is unrotatable forever.

Output: real SUPublicEDKey string committed to `MoleBar/Info.plist`, GitHub Actions secret populated, dev Mac Keychain seeded, on-disk export shredded.

Critical: this plan is `autonomous: false` because every step is a user-machine or GitHub-UI operation that Claude cannot perform. The plan provides rigorous step-by-step instructions; the user executes; Claude verifies via gh CLI / xcodebuild after the user signals completion.
</objective>

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Dev Mac login Keychain → user-space processes | The EdDSA private key is gated by Keychain ACLs; Sparkle's `generate_keys -x` requires user authorization on first export. |
| Dev Mac → /tmp/sparkle.key (transient) | The export tempfile lives ~30 seconds (between `generate_keys -x` and the GH-secret-set + shred). Mode 0600 via umask 077. |
| /tmp/sparkle.key → GitHub.com (one-time) | `gh secret set` uploads the value via HTTPS; GitHub stores it encrypted at rest. |
| GitHub.com → CI runner (per build) | Plan 06's `release.yml` reads the secret at runtime, materializes a tempfile with `umask 077`, signs the DMG, shreds the file. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|

The key is in the user's login Keychain regardless — that's the canonical store, not /tmp. |
