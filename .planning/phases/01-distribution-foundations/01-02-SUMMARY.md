## Deviations from the plan

### Deviation 1 — `@MainActor` on `CheckForUpdatesViewModel` (Swift 6 strict concurrency)

**What:** Plan task 2 prescribed an `ObservableObject` VM that subscribes to `updater.publisher(for: \.canCheckForUpdates)` from a non-isolated `init`. Under Swift 6 (locked via `SWIFT_VERSION = "6.0"`), this fails to compile:

```
error: cannot form key path to main actor-isolated property 'canCheckForUpdates'
```

`SPUUpdater.canCheckForUpdates` is main-actor-isolated in Sparkle 2.9.1. The fix is one annotation:

```swift
@MainActor
final class CheckForUpdatesViewModel: ObservableObject { ... }
```

SwiftUI `View.init` is implicitly main-actor in Swift 6, so the call-site (`CheckForUpdatesView.init(updater:)`) satisfies the constraint with no further changes. Behavior is unchanged.

The plan's prescribed code came from Sparkle's programmatic-setup docs which predate Swift 6. **Carry-forward:** Phase 2+ Swift code that touches Sparkle's KVO surface needs the same annotation; flag for `RESEARCH.md` updates if other plans reference this pattern.

### Deviation 2 — Task 2 verify regex was line-anchored, code is multi-line

**What:** The plan's Task 2 verify shell command included `grep -q 'SPUStandardUpdaterController(startingUpdater: true' MoleBar/MoleBarApp.swift`. This requires both tokens on the same line. The plan's *own* Pattern-1 code in §`<action>` writes the constructor across four lines (one argument per line). Functional code matches the plan exactly; the verify regex is brittle. Confirmed semantically with a multi-line-aware check (`tr '\n' ' ' | grep -qE 'SPUStandardUpdaterController\(\s*startingUpdater: true'`).

No code change; flagged so future plan revisions tighten verify regex literals.
